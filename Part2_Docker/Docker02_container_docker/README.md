# Docker02 Container and Docker

## 컨테이너란?

컨테이너는 애플리케이션 실행에 필요한 파일, 라이브러리, 설정을 하나의 실행 단위로 묶고 격리해서 실행하는 기술입니다. VM처럼 운영체제 전체를 새로 띄우는 것이 아니라, 호스트 OS의 커널을 공유하면서 프로세스와 파일시스템, 네트워크 등을 격리합니다.

그래서 컨테이너는 VM보다 시작 속도가 빠르고 이미지 크기도 작습니다. 개발자는 애플리케이션과 의존성을 이미지로 묶어 배포하고, 운영자는 같은 이미지를 여러 환경에서 반복 실행할 수 있습니다.

## VM과 컨테이너 비교

| 구분 | VM | Container |
| --- | --- | --- |
| 격리 방식 | 하이퍼바이저 위에 Guest OS 실행 | 호스트 커널을 공유하고 프로세스 격리 |
| 시작 속도 | 상대적으로 느림 | 빠름 |
| 크기 | OS까지 포함해 큼 | 애플리케이션과 의존성 중심 |
| 밀도 | 서버당 실행 수가 적은 편 | 더 많은 컨테이너 실행 가능 |
| 사용 목적 | 강한 OS 격리, 다양한 OS 실행 | 빠른 배포, 애플리케이션 단위 실행 |

## 컨테이너 격리 기술

| 기술 | 역할 |
| --- | --- |
| Namespace | 프로세스, 네트워크, 마운트, 사용자 공간을 분리 |
| cgroup | CPU, 메모리, I/O 같은 리소스 사용량 제한 |
| chroot | 프로세스가 볼 수 있는 루트 파일시스템 범위 제한 |
| Union filesystem | 여러 이미지 레이어를 합쳐 하나의 파일시스템처럼 제공 |

## Docker 구성 요소

| 구성 요소 | 설명 |
| --- | --- |
| Docker Client | 사용자가 입력하는 `docker` 명령 |
| Docker Daemon | 이미지 빌드, 컨테이너 실행, 네트워크와 볼륨 관리 |
| Docker Image | 컨테이너 실행에 필요한 읽기 전용 템플릿 |
| Docker Container | 이미지를 실행한 프로세스 인스턴스 |
| Dockerfile | 이미지를 만드는 절차를 코드로 적은 파일 |
| Registry | 이미지를 저장하고 가져오는 저장소 |

## Image와 Container

이미지는 실행 전의 패키지입니다. 읽기 전용 레이어들로 구성되어 있고, 같은 이미지를 여러 번 실행하면 여러 컨테이너가 만들어집니다.

컨테이너는 이미지 위에 쓰기 가능한 얇은 레이어를 추가한 실행 상태입니다. 컨테이너 내부에서 파일을 바꾸면 그 변경은 컨테이너 레이어에 기록됩니다. 컨테이너를 삭제하면 이 변경도 사라질 수 있으므로, 보존해야 하는 데이터는 volume에 저장해야 합니다.

## Registry

Registry는 이미지를 저장하는 공간입니다. 대표적으로 Docker Hub가 있고, AWS에서는 Amazon ECR을 사용합니다. 개발자가 이미지를 빌드한 뒤 registry에 push하면, 서버나 오케스트레이션 플랫폼은 그 이미지를 pull해서 컨테이너를 실행합니다.

## 기본 확인 명령

```bash
docker --version
docker compose version
docker info
docker images
docker ps -a
```

## 실습 기록

로컬 환경에서 Docker CLI와 Compose 설치는 확인했습니다. Docker daemon은 현재 실행 중이 아니어서 `docker ps`는 연결 실패가 발생했습니다. 컨테이너 실행은 Docker Desktop 또는 EC2 Docker host에서 다시 수행할 수 있도록 명령 중심으로 정리했습니다.

| 확인 항목 | 결과 |
| --- | --- |
| Docker CLI 설치 | 확인 |
| Docker Compose 설치 | 확인 |
| Docker daemon | 미실행 |
| 컨테이너 실행 | 문서화 |
