# Docker01 Cloud Native

## 핵심 개념

클라우드 네이티브는 단순히 서버를 클라우드에 올리는 것을 뜻하지 않습니다. 애플리케이션을 처음부터 클라우드의 장점인 빠른 배포, 탄력적 확장, 장애 복구, 자동화를 활용할 수 있도록 설계하는 접근입니다.

전통적인 시스템은 물리 서버나 VM 위에 하나의 큰 애플리케이션을 배포하는 경우가 많았습니다. 이 방식은 처음에는 단순하지만, 기능이 많아질수록 빌드와 배포가 느려지고 작은 변경도 전체 시스템에 영향을 줄 수 있습니다. 클라우드 네이티브는 이런 한계를 줄이기 위해 애플리케이션을 작은 서비스로 나누고, 컨테이너와 자동화된 배포 파이프라인을 사용합니다.

## 클라우드와 클라우드 네이티브의 차이

| 구분 | 클라우드 사용 | 클라우드 네이티브 |
| --- | --- | --- |
| 목적 | 인프라를 클라우드에서 사용 | 애플리케이션 구조 자체를 클라우드에 맞춤 |
| 배포 단위 | VM, 서버, 전체 애플리케이션 | 컨테이너, 마이크로서비스 |
| 변경 방식 | 수동 배포 또는 큰 단위 배포 | CI/CD 기반의 작은 단위 배포 |
| 확장 방식 | 서버 단위 확장 | 서비스 단위 확장 |
| 장애 대응 | 서버 복구 중심 | 장애 격리와 자동 복구 중심 |

## 클라우드 네이티브 구성 요소

| 요소 | 의미 |
| --- | --- |
| Container | 애플리케이션과 실행 환경을 하나의 이미지로 패키징 |
| Microservice | 큰 애플리케이션을 독립 배포 가능한 작은 서비스로 분리 |
| DevOps | 개발과 운영을 분리하지 않고 함께 자동화와 품질을 관리 |
| CI/CD | 코드를 자주 통합하고 테스트, 빌드, 배포를 자동화 |
| Immutable Infrastructure | 서버를 직접 수정하지 않고 새 이미지나 새 배포 단위로 교체 |
| Declarative API | 원하는 상태를 선언하고 플랫폼이 그 상태로 맞추게 함 |
| Observability | 로그, 메트릭, 추적으로 시스템 상태를 관찰 |

## Monolithic과 MSA

Monolithic은 하나의 애플리케이션 안에 화면, 비즈니스 로직, 데이터 접근 코드가 함께 들어 있는 구조입니다. 구조가 단순하고 초기 개발이 빠르지만, 규모가 커지면 특정 기능만 수정해도 전체 애플리케이션을 다시 빌드하고 배포해야 합니다.

MSA는 애플리케이션을 업무 기능 기준으로 나누고, 각 서비스를 API로 연결하는 구조입니다. 주문, 결제, 회원, 배송 같은 기능을 독립 서비스로 분리하면 각 서비스가 독립적으로 배포되고 확장될 수 있습니다.

| 구분 | Monolithic | Microservice |
| --- | --- | --- |
| 배포 단위 | 전체 애플리케이션 | 개별 서비스 |
| 확장 단위 | 전체 시스템 | 필요한 서비스 |
| 장애 영향 | 전체 장애로 번질 수 있음 | 서비스 단위로 격리 가능 |
| 데이터 관리 | 하나의 DB를 공유하는 경우가 많음 | 서비스별 DB 또는 스키마 분리 |
| 운영 난이도 | 초기에는 쉬움 | 분산 시스템 관리가 필요 |

## 장점

- 서비스별 독립 배포가 가능해 릴리스 속도가 빨라집니다.
- 트래픽이 많은 서비스만 따로 확장할 수 있어 비용 효율이 좋아집니다.
- 장애가 특정 서비스에 머물도록 설계할 수 있습니다.
- 컨테이너 이미지를 사용하면 개발, 테스트, 운영 환경 차이를 줄일 수 있습니다.
- CI/CD를 통해 반복적인 빌드와 배포를 자동화할 수 있습니다.

## 주의할 점

- 서비스가 많아지면 네트워크 호출, 인증, 로깅, 추적이 복잡해집니다.
- 분산 트랜잭션과 데이터 정합성을 설계해야 합니다.
- 컨테이너 이미지 보안, secret 관리, 취약점 스캔이 중요해집니다.
- 클라우드 서비스에 강하게 의존하면 다른 환경으로 이동하기 어려울 수 있습니다.

## Docker와의 연결

Docker는 클라우드 네이티브의 핵심 도구 중 하나입니다. 컨테이너 이미지는 애플리케이션 실행 환경을 코드처럼 관리하게 해주고, 어디서 실행해도 비슷한 결과를 만들 수 있게 합니다. 이후 ECS, EKS, Kubernetes 같은 오케스트레이션 도구는 이 컨테이너를 여러 서버 위에서 자동 배치하고 운영합니다.
